# Development Notes - IP Geolocation Service

## 1. Implementation walkthrough

The service is built as a thin wrapper around a third-party geolocation provider:

1. **Project skeleton.** Defined the directory layout
   (`core / models / providers / routers`) and the contract between layers.
   The key abstraction is `GeoProvider` (ABC) with a `locate(ip) -> GeolocationResponse`
   method and a shared `validate_ip`.
2. **Pydantic response and error models.** `GeolocationResponse` with required
   and optional nullable fields. A single `ErrorResponse` envelope.
3. **First implementation - `ip-api.com`** (no key, IPv4+IPv6). Async-httpx,
   handling of timeouts/429/5xx, normalisation of the upstream payload to our
   schema. Picked first because it does not require registration.
4. **Domain exceptions and centralised exception-handlers.** Providers raise
   `LocationNotFoundError` / `RateLimitError` / `ProviderUnavailableError`,
   handlers in `app/main.py` convert them into HTTP responses. Providers and
   routers know nothing about HTTP status codes.
5. **Lifespan + DI.** `httpx.AsyncClient` is created once per application
   lifetime, lives on `app.state`, and is closed on shutdown.
6. **Endpoints.** `GET /v1/geo/{ip}` and `GET /v1/geo/me`.
7. **Configuration.** `pydantic-settings` + `.env`. Variables:
   `GEO_PROVIDER`, `GEO_API_KEY`, `HTTP_TIMEOUT`, `LOG_LEVEL`.
8. **Tests.** Unit tests for the provider and integration tests for the router
   via `ASGITransport`. The `check_flow.py` smoke test exercises the service
   end-to-end.
9. **Docker.** Single-stage Dockerfile + `.dockerignore`. Deliberately simple.
10. **Documentation.** README was filled in as I went.

## 3. Challenges & solutions

### 3.1. ipapi.co blocks server-side calls without an API key

**Symptom.** When running the smoke test with `GEO_PROVIDER=ipapi_co`, all
happy-path requests returned HTTP 429 with the body
`"Please contact us for a trial account or sign up for a paid plan..."`.
The same URL opened in a browser worked fine.

**Diagnostics.** Wrote a small probe script that tried several different
`User-Agent` values - all were blocked the same way. I concluded that ipapi.co
deliberately requires an API key for server-side access; the free tier only
works from a browser.

**Resolution.** Kept `ipapi_co` as the second provider with optional
`GEO_API_KEY` support. Left `ip_api` as the default - it works without
authentication. Requested a trial key from ipapi.co (the process is an email
request). The provider implementation itself is ready.

**Takeaway.** Before picking a third-party service, you should empirically
verify at least one request with a production-like client.

### 3.2. MaxMind as "phantom functionality"

Initially I added a third provider on top of MaxMind to demonstrate the
pluggable architecture. But I never actually ran it - I had no `.mmdb` file,
and the mock-based tests were not verifying anything meaningful. I decided to
delete it: unverified code is better left out.

### 3.3. Pydantic schema and "honest nulls"

Originally all fields on `GeolocationResponse` were required and providers
substituted `""`/`0.0` for missing data. I caught this on review:
`{"city": ""}` is indistinguishable to a client from `{"city": "Genuinely
Empty City"}`, and the coordinates `(0.0, 0.0)` correspond to a real point on
the map. I made the optional fields `nullable=True, default=None` and removed
all such fallbacks from the providers.

### 3.4. Duplicate `mock_ip_api` fixture

I noticed that the same fixture was defined in both `tests/conftest.py` and
`tests/test_geo_router.py` (with different `assert_all_called` settings). The
local fixture shadowed the shared one, so the conftest version was effectively
dead code. I removed `conftest.py` entirely.

## 4. GenAI usage

The development approach was **AI-heavy**: most of the code and tests were
generated by assistants (GitHub Copilot, Sonnet 4.6 and Opus 4.7 models).

**My role - tech lead / code reviewer:**

- set tasks and agreed on the plan;
- critically read the proposed code, requested fixes and simplifications;
- made architectural decisions and defined scope (what is in, what is out);
- did real, empirical checks against the upstream services;
- dropped unverified functionality;
- managed commits.

**Where AI helped well:** Pydantic models, scaffolding, test drafts with
mocks, docstrings in a consistent style, edge cases.

**Where I had to push back:** a tendency towards over-engineering (it
suggested structured logging, healthchecks, caching - I had to cut these),
and producing code that looked plausible but was not actually verified.

**Conclusion:** AI sped up typing by a large margin, but it required constant
oversight from the developer.

## 5. API Design Decisions

- REST, version in the URL (`/v1`).
- Two GET endpoints: `/v1/geo/{ip}` and `/v1/geo/me`. Both return the same
  response schema.
- Optional fields come back as `null`, not an empty string.
- All errors share the same envelope (`error`, `message`, `detail`).
- The active provider is selected through a single environment variable.

---

## 6. Third-party API/Database selection

**Chose Option A (third-party API).** The main reason is convenience: you run
`docker run` and immediately see a working service, without downloading
licensed files or signing up anywhere.

For the API itself I picked **ip-api.com**: no key required, supports
IPv4/IPv6, returns all the needed fields in a single call, and the free
45 req/min budget is plenty for a demo.

I kept the second provider, `ipapi_co`, **to demonstrate the abstraction** -
switching providers is a single environment variable. It also shows how an
optional API key plugs into the design.

**For production** I would most likely go with a local database (MaxMind
GeoLite2): fast, no external dependency risk, no rate limits, predictable
cost. The only downside is that the `.mmdb` file has to be refreshed
periodically.

## 7. Production Readiness - next steps

### 7.1. Health checks

A standard `GET /health` for a liveness probe (k8s/Docker). One level deeper -
a `GET /ready` that additionally verifies that the provider has been
initialised.

### 7.2. Caching upstream responses

IP-to-geo data changes rarely. A Redis or in-memory cache with a TTL would
be a natural fit.

### 7.3. Retry with exponential backoff

Retry a couple of times instead of failing immediately when the provider is
temporarily unavailable.

### 7.4. Authentication / Authorization

The service is currently open - the data is public and authentication is not
strictly needed. In production I would add auth, plus an API-key middleware
if trusted internal services start consuming it.

### 7.5. MaxMind as a real alternative provider

MaxMind is currently removed because there was no way to verify it against a
real database. If it comes back, it needs an integration test with an actual
`.mmdb` file. I would also discuss provider rotation with the tech lead:
if one provider is unavailable or returns errors, switch to the other.
